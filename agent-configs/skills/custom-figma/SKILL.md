# Figma 改造分析 Skill

## 触发条件

用户消息中包含 `figma.com/design/` 或 `figma.com/file/` 链接时触发。

## 用户输入

| 参数         | 必需 | 说明                               |
| ------------ | ---- | ---------------------------------- |
| Figma URL(s) | 是   | 一个或多个 Figma 设计稿链接        |
| 组件范围区域 | 是   | 指定要对比的代码组件路径或区域描述 |
| 需求说明     | 否   | 额外的改造意图描述                 |

如果用户未提供组件范围区域，主动询问。

---

## 一、调用策略

### 1.1 URL 解析与合并

从 Figma 链接中提取 `fileKey` 和 `nodeId`：

- `fileKey`：URL 路径中 `/design/<fileKey>/` 或 `/file/<fileKey>/` 部分
- `nodeId`：URL 查询参数 `node-id` 的值，将 `-` 替换为 `:`（如 `94210-74603` → `94210:74603`）

**同 fileKey 多节点合并**：将所有 nodeId 用逗号拼接，通过 `ids` 参数一次请求：

```http
GET /v1/files/:fileKey/nodes?ids=94210:74603,94210:74607,61428:109218
```

不同 fileKey 之间分别请求，间隔至少 5 秒。

### 1.2 限流处理

遇到 Figma API 429 错误时：

1. **第 1 次重试**：`sleep 15` 后重试
2. **仍然失败**：停止重试，告知用户，用已获取的部分数据继续分析

---

## 二、分析策略

获取 Figma 数据后，按以下步骤执行分析：

### Step 1 — 读取 figma-theme

读取项目的 `web/src/pages/pollo.ai/_styles/figma-theme.ts` 文件，建立完整的颜色值 → `-f-` CSS 变量映射表。后续所有颜色对比基于此映射。

### Step 2 — 多图去重与组件识别

对比多个 Figma 节点的结构树：

- **设计稿完全一致的部分** → 确认是否使用同一代码组件，**合并为一次修改**，不要重复输出
- **结构相同但内容/样式有差异的部分** → 进入 Step 3 分析差异原因

### Step 3 — 差异分析与状态推断

当Figma 节点重复 item 中存在样式差异时，按以下优先级判断：

1. **检查 Figma componentProperties**：是否包含 Variant 值（如 `Property 1=Pro/Plus`、`Property 1=未签到/已签到`）
2. **检查 CSS 交互状态**：差异是否对应 hover / active / disabled / focus 等伪类状态
3. **检查条件渲染**：差异是否对应业务逻辑分支（如登录/未登录、有套餐/无套餐、Pro/Plus）

将差异归类为以下类型之一：

| 差异类型     | 说明                     | 代码表现                    |
| ------------ | ------------------------ | --------------------------- |
| Variant 状态 | Figma 组件的不同变体     | 条件渲染不同子组件          |
| CSS 伪类状态 | hover/active/disabled 等 | 伪类选择器或 className 切换 |
| 条件渲染分支 | 业务逻辑驱动的不同展示   | if/三元表达式/match 分支    |

### Step 4 — 布局层级对比

对比 Figma 层级结构与现有代码的 DOM/JSX 结构，识别以下变动：

- **新增/删除包裹层**：Figma 中多了或少了一层容器
- **排列方向变化**：row ↔ column（flex-direction）
- **元素顺序调整**：子元素的先后顺序改变
- **嵌套关系变化**：某元素从同级变为子级，或反之
- **间距/对齐变化**：gap、padding、justify/align 属性的变化

### Step 5 — 现有组件逻辑分析（代码逻辑优先于 Figma 变动）

**核心原则**：组件现有业务逻辑的优先级高于 Figma 设计变动。深入读取现有代码，理解而非改变业务逻辑。

#### 5.1 逻辑要素提取

- **条件分支**：现有的 if/三元/match 分支，标注每个分支的业务场景和触发条件
- **变量及数据来源**：每个变量/useMemo 的数据维度（如 plan/addon/freeReward/total）、是固定文本还是动态计算值
- **状态管理**：useState / zustand store / context 中的状态变量及其用途
- **事件处理**：onClick、onChange 等事件回调及其副作用
- **Hooks 依赖**：自定义 hooks、useEffect 副作用、useMemo/useMemoizedFn 缓存逻辑
- **Props 传递链**：父组件传入的 props 及其在子组件中的流转
- **API 调用**：tRPC 调用、数据获取与提交逻辑
- **国际化**：现有的 `<Trans>` / `t` 模板用法，动态字符串必须用 `t` 模板（不可用 `<Trans>` 包裹运行时变量）

#### 5.2 逻辑去留判定

- **无冲突 → 直接保留**，不在文档中逐条列出，仅在"保留项"章节汇总说明
- **需调整 → 标出**，体现在改造详情的「逻辑改变清单」中，说明调整原因和方式
- **存疑 → 提出**，放入「冲突与疑问」章节，等待用户决策

判定标准：

| 结果     | 含义                                         | 处理方式                         |
| -------- | -------------------------------------------- | -------------------------------- |
| **保留** | 逻辑与新设计无冲突                           | 直接保留，汇总在"保留项"章节     |
| **调整** | 逻辑保留但需适配新结构（移动位置、改参数）   | 标出，写入「逻辑改变清单」       |
| **存疑** | 设计稿未明确体现但代码中存在的隐式业务逻辑   | 写入「冲突与疑问」，等待用户决策 |

> **禁止默认删除任何业务逻辑**。即使设计稿中找不到对应 UI，也必须在「冲突与疑问」中提出，不得自行判断删除。

### Step 6 — 代码对比与改造项提取

读取用户指定范围的现有代码，逐项对比 Figma 设计与代码实现：

- 颜色值通过 figma-theme 映射为 `-f-` Tailwind class
- 硬编码颜色必须替换为 theme 变量
- 记录每一处差异作为一个改造项，给予全局编号
- **逻辑融入**：现有 hooks/state/事件处理如何与新 DOM 结构对接，标注调整方式
- **Figma 静态文本 vs 代码动态变量**：当 Figma 中展示的是固定文本（如 "Web Credits"），但代码中对应的是动态变量（如 `planName`），必须使用代码中的变量，不得硬编码 Figma 文本

### Step 7 — 生成改造分析文档

将以上分析结果整理为改造分析文档：

- **文档位置**：根据上下文判断合适的位置，默认为项目根目录
- **文档内容**：按下方「输出格式」章节的结构生成
- **lint 修复**：在文档末尾附上项目的 eslint fix 命令，代码修改完成后必须执行

## 三、输出格式：改造分析文档

### 整体结构

```markdown

- 共 N 个 Figma 节点
- 节点 A (nodeId) → 对应状态/变体描述
- 节点 B (nodeId) → 对应状态/变体描述
- 去重结果：节点 A 和节点 C 结构一致，共用组件 XxxComponent

## 改造项列表

| #   | 改造名称 | 涉及文件          | 差异类型 |
| --- | -------- | ----------------- | -------- |
| 1   | xxx      | path/to/file.tsx  | 条件渲染 |
| 2   | xxx      | path/to/file.tsx  | 样式调整 |
| 3   | xxx      | path/to/other.tsx | 布局变动 |

## 改造详情

### path/to/file.tsx

（该文件下的所有改造项 diff）

(该文件下的含业务逻辑改变清单(可选))

### path/to/other.tsx

（该文件下的所有改造项 diff）

(该文件下的含业务逻辑改变清单(可选))

## 冲突与疑问(可选)

> 以下为分析过程中发现的设计稿与现有代码之间的冲突或无法自动判断的决策点，**需要用户确认后方可执行**。

| #   | 类型 | 描述                                     | 现有代码行为       | 设计稿表现       | 建议处理方式            |
| --- | ---- | ---------------------------------------- | ------------------ | ---------------- | ----------------------- |
| Q1  | 冲突 | 设计稿移除了 xx 区域，但代码中有关联逻辑 | 点击触发 xx API    | 该按钮不存在     | 建议删除 / 移至其他位置 |
| Q2  | 存疑 | 设计稿未体现 xx 状态，但代码中有条件渲染 | 根据 isXx 显示不同 | 只展示了一种状态 | 待确认是否保留          |
```

### Diff 代码块格式规范

每个文件一个 diff 代码块，文件内用注释分隔不同改造项：

````markdown
```diff
// /absolute/path/to/file.tsx

  // --- 改造 #1: 改造名称 ---
  <div className="existing">                          // L86 现有容器
-   <span className="text-gray-500">old</span>        // 旧: 使用了默认灰色
+   <span className="text-f-text-secondary">new</span>// 改为 figma-theme 次要文字色
    ...
  </div>

  // --- 改造 #2: 改造名称 ---
  <Button                                             // L109 操作按钮
-   className='mt-4 h-10 w-full'                      // 旧间距 mt-4
+   className='mt-2 h-10 w-full'                      // 缩小间距，对齐设计稿
  >
```
````

**格式规则：**

1. **文件路径**：第一行为绝对路径，不带行号
2. **改造项分隔**：用 `// --- 改造 #N: 名称 ---` 注释分隔同文件内的多个改造项
3. **行号标注**：上下文行右侧注释标注 `// L行号 元素描述`
4. **修改注释**：每行 `-`/`+` 行右侧注释标注**修改效果**（而非修改内容）
5. **省略标记**：未变动的中间行用 `...` 省略，保留足够上下文行辅助定位
6. **新增代码块**：用 `+` 标记每一行，右侧注释说明该行作用
7. **删除代码块**：用 `-` 标记每一行，右侧注释说明为何删除

## 四、注意事项

1. **禁止使用 Tailwind 默认色**：所有颜色必须映射到 figma-theme 的 `-f-` 变量
2. **合并同组件修改**：多个 Figma 节点指向同一代码组件时，只输出一份 diff，在注释中说明适用于哪些状态
3. **状态判断要有依据**：标注差异类型时，必须说明判断依据（Figma Variant 值 / CSS 伪类规则 / 业务逻辑）
4. **保持改造项编号全局唯一**：跨文件的改造项使用同一编号序列
5. **先输出文档，等用户同意后再执行修改**：改造分析文档仅为计划，不自动执行代码修改
6. **保留业务逻辑优先**：当设计稿的样式/布局变更与现有业务逻辑冲突时，优先保留业务逻辑，在「冲突与疑问」中提出供用户决策
7. **隐式逻辑必须提出疑问**：设计稿中未明确体现但代码中存在的逻辑（如埋点、错误处理、权限校验、副作用），不得默认删除，必须在「冲突与疑问」中列出
