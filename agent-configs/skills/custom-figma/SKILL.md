# Figma 改造分析 Skill

## 触发条件

用户消息中包含 `figma.com/design/` 或 `figma.com/file/` 链接时触发。

## 用户输入

| 参数              | 必需 | 说明                                             |
| ----------------- | ---- | ------------------------------------------------ |
| Figma URL(s)      | 是   | 一个或多个 Figma 设计稿链接                      |
| 组件范围区域      | 是   | 指定要对比的代码组件路径或区域描述               |
| figma数据采用范围 | 是   | figma数据采用范围描述(如：样式 布局，不包含文本) |
| 需求说明          | 否   | 额外的改造意图描述                               |

如果用户未提供组件范围区域，主动询问。

---

## 一、调用策略

本 skill 通过 **Framelink Figma MCP**（npm 包 `figma-developer-mcp`，配置名 `Framelink-Figma-MCP`，token 从环境变量 `FIGMA_MCP_API_KEY` 读取）与 Figma 交互。

Figma API 返回任何错误（403、429、500 等）时，**立即停止流程**，将错误信息告知用户。

---

## 二、分析策略

获取 Figma 数据后，按以下步骤执行分析。**先理解代码，代码逻辑优先于 Figma 变动，再解读设计稿，最后对比提取改造项。**

### Step 1 — 组件分析

**核心原则**：先理解代码再看设计稿。组件现有业务逻辑的优先级高于 Figma 设计变动。

从以下维度分析组件：

- **DOM/JSX 结构**：组件的层级嵌套、容器划分、子组件组合方式
- **条件分支**：现有的 if/三元/match 分支，标注每个分支的业务场景和触发条件
- **变量及数据来源**：每个变量/useMemo/useState 的数据维度（如 plan/addon/freeReward/total）、是固定文本还是动态计算值
- **国际化**：识别 `<Trans>` / `t` 模板的使用方式及动态变量

### Step 2 — Figma 数据分析

从多个维度解析 Figma 节点数据，可结合 Step 1 的组件分析辅助理解：

- **层级结构**：识别 Frame/Group 的嵌套关系、auto-layout 方向（horizontal/vertical）、间距（itemSpacing）、padding、子节点排列顺序
- **状态与变体**：检查 componentProperties 的 Variant 值、识别 hover/active/disabled/focus 等伪类状态对应的样式差异
- **多节点去重**：对比多个 Figma 节点的结构树，完全一致的部分合并，有差异的部分标记差异原因
- **样式提取**：颜色填充/描边/阴影的 RGBA 值、字号/字重/行高、宽高/圆角/padding/gap

### Step 3 — 属性级逐项对比

1. 先读取项目的 `web/src/pages/pollo.ai/_styles/figma-theme.ts` 文件，建立颜色值 → `-f-` CSS 变量映射表。
2. 根据用户指定的 figma 数据采用范围，过滤不适用的对比维度
3. 基于 Step 1（组件分析）与 Step 2（Figma 数据）的分析结果，**对每个现有 UI 元素执行属性级 checklist 对比**：

**高频遗漏点**

| 遗漏类型       | 典型案例                                   | 检查方法                                          |
| -------------- | ------------------------------------------ | ------------------------------------------------- |
| **图标方向**   | com-right 应改为 com-down/com-up           | 对比 Figma componentId 与代码 icon class          |
| **圆角差异**   | rounded-full(圆) vs rounded-lg(8px 方圆角) | 对比 Figma borderRadius 与代码 rounded-\* class   |
| **可见性变化** | hover 才显示 → 始终显示                    | 检查 Figma 中元素是否有条件隐藏，对比代码交互逻辑 |
| **布局方向**   | hover 弹出侧面板 → 点击展开内联列表        | 对比 Figma 子节点排列方式与代码实现方式           |
| **固定尺寸**   | w-[360px] 应为 md:w-[360px]                | 每个固定尺寸值都检查：移动端是否需要自适应        |
| **对齐方式**   | 图标/按钮应靠右但未生效                    | 检查 justify-between 是否搭配 w-full              |

#### 3.1 改造项提取规则

- 每处 Figma 与代码的差异作为一个改造项，给予全局唯一编号
- 按文件分组，同文件的改造项归入同一个 diff 代码块
- 标注改造类型（样式调整 / 布局变动 / 新增元素 / 条件渲染 / 代码清理）
- 涉及逻辑调整的改造项，附「逻辑改变清单」说明调整原因

#### 3.2 逻辑去留判定

对比过程中，对 Step 1 提取的每个逻辑要素进行去留判定：

- **无冲突 → 直接保留**，不在文档中逐条列出，仅在"保留项"章节汇总说明
- **需调整 → 标出**，体现在改造详情的「逻辑改变清单」中，说明调整原因和方式
- **存疑 → 提出**，放入「冲突与疑问」章节，等待用户决策

判定标准：

| 结果     | 含义                                       | 处理方式                         |
| -------- | ------------------------------------------ | -------------------------------- |
| **保留** | 逻辑与新设计无冲突                         | 直接保留，汇总在"保留项"章节     |
| **调整** | 逻辑保留但需适配新结构（移动位置、改参数） | 标出，写入「逻辑改变清单」       |
| **存疑** | 设计稿未明确体现但代码中存在的隐式业务逻辑 | 写入「冲突与疑问」，等待用户决策 |

### Step 4 — 生成改造分析文档

将以上分析结果整理为改造分析文档：

- **文档位置**：根据上下文判断合适的位置，默认为项目根目录
- **文档内容**：按下方「输出格式」章节的结构生成

### Diff 代码块格式规范

每个文件一个 diff 代码块，文件内用注释分隔不同改造项：

````markdown
```diff
// /absolute/path/to/file.tsx

  // --- 改造 #1: 改造名称 :86 ---
  // --- 决策理由 ---
  <div className="existing">                          // 现有容器
-   <span className="text-gray-500">old</span>        // 旧: 使用了默认灰色
+   <span className="text-f-text-secondary">new</span>// 改为 figma-theme 次要文字色
    ...
  </div>
    ...
  // --- 改造 #2: 改造名称 :104 ---
  // --- 决策理由 ---
  <Button                                             // 操作按钮
-   className='mt-4 h-10 w-full'                      // 旧间距 mt-4
+   className='mt-2 h-10 w-full'                      // 缩小间距，对齐设计稿
  >
```
````

**格式规则：**

1. **文件路径**：第一行为绝对路径，不带行号
2. **改造项分隔**：用 `// --- 改造 #N: 名称 :行号 ---` 注释分隔同文件内的多个改造项 并标注`:行号`
3. **决策理由**：`改造项分隔` 下方紧跟决策理由
4. **行注释**： `-`/`+` 须右侧注释 修改说明
5. **省略标记**：未变动的中间行用 `...` 省略，保留足够上下文辅助定位

## 四、注意事项

### 颜色与样式

1. **禁止使用 Tailwind 默认色**：禁止硬编码 hex 值或使用 gray/zinc/slate 等默认色，所有颜色必须通过 figma-theme 映射为 `-f-` CSS 变量

### 变量与数据

2. **变量优先于文本**：Figma 中的固定文本若对应代码中的动态变量（如 `planName`、`creditsValue`），必须使用变量，不可替换为硬编码文本
3. **禁止替换变量的数据来源**：不得自行更改变量所引用的数据字段（如将 `planUseCount` 替换为 `nonPlanUseCount`），必须保持原有数据映射关系

### 国际化

4. **翻译方案选择**：静态文本用 `<Trans>`，包含运行时变量的动态字符串必须用 `t` 模板（不可用 `<Trans>` 包裹运行时变量，否则翻译会失败）

### 业务逻辑保护

5. **简化条件分支必须提出疑问**：代码中有多条件分支，对应 Figma 展示的只有单一状态时，不得默认删除，必须在「冲突与疑问」中列出
6. **保留业务逻辑优先**：当设计稿的样式/布局变更与现有业务逻辑冲突时，优先保留业务逻辑，在「冲突与疑问」中提出供用户决策
7. **隐式逻辑必须提出疑问**：设计稿中未明确体现但代码中存在的逻辑（如埋点、错误处理、权限校验、副作用），不得默认删除，必须在「冲突与疑问」中列出

### 输出规范

8. **状态判断要有依据**：标注差异类型时，必须说明判断依据（Figma Variant 值 / CSS 伪类规则 / 业务逻辑）
9. **先输出文档，等用户同意后再执行修改**：改造分析文档仅为计划，不自动执行代码修改
