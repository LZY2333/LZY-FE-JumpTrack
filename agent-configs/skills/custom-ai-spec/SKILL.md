## 核心原则

- **只写 WHAT 和 WHY，不写 HOW** — 不涉及技术选型、代码结构、文件路径
- **非技术人员也能看懂** — 产品经理/设计师可审阅
- **以用户故事为核心** — 每个故事独立可测、有优先级
- **spec 是活的文档** — 是 AI 和开发者的共享信息源，不是一次性产物

## 工作流

### 步骤 1：接收输入

- 如果用户在命令后**已带输入内容**（文字、截图、链接等），直接作为需求描述使用
- 如果**无输入**，输出以下建议后等待用户输入：

```
请描述你的需求，例如：
- 一句话需求："用户头像支持裁剪上传"
- 指定模块："修改 Header 组件的导航菜单"
- 贴设计稿截图
- 贴 PRD 链接或文档内容
```

### 步骤 2：理解项目上下文(package.json, 项目目录结构, 必要的具体代码)

在分析需求前，先快速了解项目现状：

1. 读取 `CLAUDE.md`（如存在），了解项目约定和技术栈
2. 读取 `package.json`（如存在），了解依赖和脚本命令
3. 浏览项目根目录结构，理解项目组织方式

> 目的不是扫描具体代码，而是让 spec 中的需求描述**契合项目的现有语境**。

### 步骤 3：分析需求

1. 解析用户输入，提取所有需求点（**不得遗漏任何一条**）
2. 将需求组织为**用户故事**，每个故事独立可测
3. 为每个用户故事分配优先级（P1 = MVP 必须, P2 = 重要, P3 = 锦上添花）
4. 识别涉及的**关键实体**（数据对象及其关系，不涉及技术实现）
5. 对不确定的内容：
   - 影响范围大的标记为 `[待澄清]`（**最多 3 个**）
   - 其余用行业惯例/合理默认值处理，记录在「假设与默认值」段

### 步骤 4：生成文档

- 在项目根目录创建 `plan/` 目录（如不存在）
- 按下方模板生成 `plan/0.AI-SPEC.md`

### 步骤 5：质量自检

生成后，对照以下 checklist 自动验证：

- [ ] 不含实现细节（无框架名、API 路径、代码结构）
- [ ] 每个用户故事有完整的验收标准（Given/When/Then）
- [ ] 成功标准可量化、技术无关
- [ ] 所有功能需求可测试、无歧义
- [ ] 边界条件已识别
- [ ] 待澄清项 ≤ 3 个
- [ ] 每个需求可追溯到用户原始输入

如有不通过项，修正后重新检查（最多迭代 3 次）。

### 步骤 6：输出

- 输出完成提示
- 如有待澄清项，以选项表格形式呈现，等待用户选择
- 告知用户审核并修改后运行 `/ai-plan` 生成执行计划

## 写作规范

- 非技术内容使用中文，技术术语、代码标识符保留原文
- **不含执行细节**（怎么改、改哪行、用什么库），只描述"做什么"和"为什么"
- 每个功能需求标注来源用户故事（如 `来源：US1`），确保可追溯
- 不得遗漏用户输入中的任何需求点
- 待澄清项最多 3 个，其余用合理默认值处理并记录在「假设与默认值」
- 验收标准必须具体、可测试，避免 "应该正常工作" 等模糊表述
- 成功标准必须**可量化、技术无关**（不能写"API 响应 < 200ms"，要写"用户感知即时响应"）
- 功能约束的三层边界要**针对本次功能**，不是泛泛的通用规则
