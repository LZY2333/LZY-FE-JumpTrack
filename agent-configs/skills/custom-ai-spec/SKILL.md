## 核心原则

- **只写 WHAT 和 WHY，不写 HOW**
- 不涉及技术实现、代码结构、文件路径
- 不扫描代码（代码扫描在 `/ai-plan` 阶段执行）
- 产品经理/设计师也能看懂

## 工作流

### 步骤 1：接收输入

- 如果用户在命令后**已带输入内容**（文字、截图、链接等），直接作为需求描述使用
- 如果**无输入**，输出以下建议后等待用户输入：

```
请描述你的需求，例如：
- 一句话需求："用户头像支持裁剪上传"
- 指定模块："修改 Header 组件的导航菜单"
- 贴设计稿截图
- 贴 PRD 链接或文档内容
```

### 步骤 2：分析需求

1. 解析用户输入，提取所有需求点（不得遗漏任何一条）
2. 为每个需求点分配优先级（P1/P2/P3）
3. 对不确定的内容：
   - 影响范围大的标记为 `[待澄清]`（**最多 3 个**）
   - 其余用行业惯例默认，记录在「假设与默认值」段
4. 将分析结果结合下方模板，生成 `plan/0.AI-SPEC.md`

### 步骤 3：输出

- 在项目根目录创建 `plan/` 目录（如不存在）
- 生成 `plan/0.AI-SPEC.md`
- 输出完成提示，告知用户审核并修改后运行 `/ai-plan` 生成执行计划

## 写作规范

- 非技术内容使用中文
- 技术术语、代码标识符保留原文
- **不含执行细节**（怎么改、改哪行、用什么库），只描述"做什么"和"为什么"
- 每个需求点必须可追溯到用户原始输入
- 不得遗漏用户输入中的任何需求点
- 待澄清项最多 3 个，其余用合理默认值处理并记录在「假设与默认值」
- 验收标准必须具体、可测试，避免模糊表述
