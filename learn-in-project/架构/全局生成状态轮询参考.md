# 全局生成状态轮询参考

目标：给当前项目新增“用户作品开始生成后，统一进入全局轮询；任务完成时触发提醒”的能力。

本文只整理项目中已经存在、最接近该目标的实现，按 3 部分归档：

1. 全局挂载，并且可调用的功能
2. 轮询
3. 查询生成状态的接口

---

## 1. 全局挂载，并且可调用的功能

### 1.1 Next App 全局挂载点

参考文件：`web/src/pages/_app.page.tsx`

关键点：
- 全局脚本、全局 Provider、全局异常处理都统一挂在这里。
- 如果要新增“全局生成状态管理器”，这是最稳定的根挂载点之一。

参考位置：
- `CoreDataScript / ABTestScript / PageWaitingScript / UserPreferenceScript`
- `SiteContextProvider`
- `AppSessionProvider`
- `AppUIModalProvider`
- `GlobalErrorHandler`
- `DynamicGlobalComponent`

适合借鉴：
- 把一个“无 UI 但全站生效”的组件挂在最外层。
- 保证切页后依然存在，不依赖某个业务页组件是否挂载。

### 1.2 Pollo 站点根布局里的全局逻辑挂载

参考文件：`web/src/pages/pollo.ai/_layout/layouts/RootLayout/index.tsx`

关键点：
- 这里已经存在“根布局挂一次，然后持续工作”的模式。
- `useCheckInNotification()` 直接在根布局执行。
- `<PolloGlobalEvent />` 也是一个无 UI 的全局监听组件。

适合借鉴：
- 如果功能只需要在 `pollo.ai` 站点内生效，把全局生成轮询组件挂在这里更合适。
- 比直接散落在页面级组件中更稳定。

### 1.3 全局 Context Provider 模式

参考文件：`web/src/contexts/SiteSpecificContext/index.ts`

关键点：
- `SiteSpecificContextProvider` 在全局挂载。
- 内部 `useSiteSpecific()` 聚合多个功能 hook：`useUserInfo / useSubLog / useSubUsage`。
- 业务侧通过 `useSiteSpecificContext()` 统一调用。

适合借鉴：
- 如果要做“可调用的全局功能”，可以做成 `GenerationStatusContextProvider + useGenerationStatusContext()`。
- Provider 负责：轮询队列、开始/结束注册、去重、清理、提醒。
- 页面/表单/生成按钮只调用 `registerGenerationTask()` 之类的方法即可。

### 1.4 全局 hook 在 Context 内自动请求和处理副作用

参考文件：`web/src/contexts/SiteSpecificContext/hooks/sub-usage/index.ts`

关键点：
- `useSubUsage()` 本身就是“全局挂载后自己请求接口”的例子。
- 内部直接 `api.xxx.useQuery(...)`。
- `onSuccess` 中继续触发副作用：`showFreeCreditsNotification(data)`。

适合借鉴：
- 全局生成状态管理器也可以用同样思路：
  - Provider 内部维护任务列表
  - 内部发起轮询请求
  - 请求成功后判断是否完成
  - 完成后调用 `message.success` / `notification.open`

### 1.5 全局事件总线

参考文件：
- `web/src/events/index.ts`
- `web/src/pages/pollo.ai/_components/PolloGlobalEvent/events/index.ts`

关键点：
- `globalEventEmitter` 是全局事件总线。
- `PolloGlobalEvent` 这种组件会在全局挂载后，统一监听某类事件，再触发业务回调。

适合借鉴：
- 如果不想让生成按钮直接依赖 Context，也可以走事件总线：
  - 开始生成时 emit
  - 全局组件监听后写入轮询队列

### 1.6 最贴近“生成开始时触发全局功能”的现有入口

参考文件：
- `web/src/utils/api.ts`
- `web/src/utils/generation-task-link.ts`
- `web/src/pages/pollo.ai/_components/Creations/events.ts`

关键点：
- `generationTaskLink` 被挂在 tRPC 全局 client links 里。
- 它会拦截一批生成相关 mutation。
- 在生成请求发起时 emit `creationGenerate: started`。
- 在成功/失败时继续 emit `success / failed`。

这是当前项目里最像“全局可调用入口”的现成实现。

适合借鉴：
- 不需要在每个生成页面重复埋注册逻辑。
- 直接在 `generationTaskLink` 里接入“开始轮询”的注册动作，改造成本最低。
- 目前它已经覆盖了视频生成、视频工具、图片生成、图片工具的多条 path。

建议优先级：
1. `generationTaskLink` 作为统一开始入口
2. `RootLayout` 作为全局挂载位置
3. `Context Provider` 或全局事件组件作为运行容器

## 2. 轮询

### 2.1 页面级通用轮询 Hook

参考文件：`web/src/hooks/useRecordsStatusMonitor.ts`

关键点：
- 输入是当前页面持有的 `myRecords`。
- 只会筛选 `Waiting / Processing` 状态。
- 每 5 秒调用一次 `generation.queryRecordDetail.fetch({ id })`。
- 提供两个回调：
  - `onFetchSuccess`
  - `onFinished`

优点：
- 逻辑简单，容易直接复用。

限制：
- 页面级，依赖 React 组件挂载。
- 组件卸载后立即停止。
- 更适合作为局部参考，不适合作为真正全局方案的最终形态。

### 2.2 Canvas 的“只能全局调用一次”轮询

参考文件：`web/src/pages/pollo.ai/home/canvas/_hooks/useCanvasPolling.ts`

这是最值得重点参考的轮询实现。

关键点：
- 注释明确写了：`只能全局调用一次，避免多次调用`。
- 使用 `useInterval + useLockFn` 控制轮询节奏。
- 内部不是只轮询视频，而是按 `mediaType` 分流处理：
  - `MediaType.image`
  - `MediaType.video`
- 每轮轮询后更新全局 store，并处理自动保存等后置逻辑。

这套模式和你的目标最接近：
- 单点挂载
- 全局运行
- 支持 image / video
- 可以在轮询完成后做统一后处理

建议复用点：
- 轮询主循环结构
- `useLockFn` 防重入
- `useInterval` 固定调度
- `match(mediaType)` 分图片/视频处理

### 2.3 Header History 的持久化轮询

参考文件：`web/src/pages/pollo.ai/_layout/Header/History/index.tsx`

关键点：
- 使用 `localStorage` 存储：
  - `local-generating-record-ids`
  - `local-generated-record-ids`
- 使用 `api.video.getVideoStatusByRecordIds.useQuery(..., { refetchInterval: 5000 })`
- 轮询完成后，从 generating 列表中移除，加入 generated 列表。

优点：
- 已经有“任务队列持久化”的思路。
- 切页后状态仍然可以继续保留。

限制：
- 当前只做 badge 计数，不做消息提醒。
- 更偏 Header 历史入口，不是抽象的全局调度器。

适合借鉴：
- localStorage 队列存储
- 批量按 recordId 查状态
- 完成后迁移队列状态

### 2.4 Agent 的集中式轮询调度器

参考文件：`web/src/pages/pollo.ai/app/ai-agent/context/AgentScheduler.ts`

关键点：
- 使用 Zustand Context 创建调度器。
- Store 内维护：
  - `tasks`
  - `timer`
  - `isPolling`
  - `registeredTools`
  - `pollingConfig`
- `registerTool()` 可动态注册待轮询对象。
- `_scheduler()` 统一启动 `setInterval`。

适合借鉴：
- 如果你想把“全局生成轮询”做成正式调度器，这个结构最规范。
- 特别适合后续要支持：
  - 注册多个任务
  - 去重
  - 统一启动/停止
  - 扩展轮询间隔策略

建议优先级：
1. `useCanvasPolling` 参考轮询主循环和 image/video 分流
2. `History` 参考 localStorage 持久化队列
3. `AgentScheduler` 参考调度器抽象和注册/反注册结构
4. `useRecordsStatusMonitor` 参考最小可用逻辑

## 3. 查询生成状态的接口

### 3.1 `generation.queryRecordDetail`

参考文件：`server/router/routes/generation.ts`

关键点：
- 入参：`id`（recordId）
- 内部调用：`getUserVideoList({ recordIds: [id] })`
- 返回：完整的 record detail

这是最通用、最值得优先考虑的接口。

原因：
- 输入是 `recordId`，和“开始生成成功后拿到 recordId 再注册轮询”天然匹配。
- 返回是完整记录，适合做：
  - 状态判断
  - 提醒文案
  - 图片/视频差异处理
  - 后续跳转/打开详情

### 3.2 `queryRecordDetail` 返回结构本身支持图片和视频

参考文件：
- `web/src/pages/pollo.ai/_types/video.ts`
- `server/interface/vo/app/video.ts`

关键点：
- 前端 `CreationItem` 来自 `video.getUserVideoList.data[number]`
- 返回结构中的：
  - `generations[].mediaType` 明确区分 image/video
  - `generateRecord.status` 是统一状态字段
  - `originalAssets[].mediaType` 也有媒体类型

结论：
- 这条接口链不是只能服务视频。
- 图片和视频已经走统一 record 结构。
- 如果你的全局轮询按 `recordId` 做，图片/视频都能覆盖。

### 3.3 `video.getVideoStatusByRecordIds`

参考文件：
- `server/router/routes/video.ts`
- `server/service/video/status.ts`

关键点：
- 输入：`recordIds: number[]`
- 返回：轻量级状态结果
  - `generateRecord`
  - `videoList`
  - `notFoundRecordIds`

这是最适合“批量轮询队列”的接口。

适用场景：
- 全局队列里同时有多个生成任务
- 不想每个 record 都单独请求一次 `queryRecordDetail`
- 优先只做状态更新，不需要完整详情

优点：
- 批量查询，天然适合轮询队列
- 现有 Header History 已经在用

限制：
- 返回较轻，不如 `queryRecordDetail` 信息全
- 如果提醒弹窗要展示更丰富信息，可能还要补查详情

### 3.4 `video.pollingTaskStatus`

参考文件：`server/router/routes/video.ts`

关键点：
- 输入：`videoIds: string[]`
- 更偏结果项维度，而不是 record 维度。

适用场景：
- 资产页、单个结果刷新
- 已经持有 `videoId`，只想刷新结果项状态

限制：
- 对“全局开始生成 -> 进入轮询队列”这条链来说，不如 `recordId` 自然。
- 因为开始生成时更稳定拿到的是 record 维度信息。

建议接口优先级：
1. `generation.queryRecordDetail`
   - 适合完整轮询和 image/video 共用
2. `video.getVideoStatusByRecordIds`
   - 适合批量队列轮询优化
3. `video.pollingTaskStatus`
   - 更适合 videoId 维度刷新，不建议作为主方案入口

## 推荐组合

如果要做一个新的“全局生成状态轮询 + 完成提醒”功能，建议优先按下面组合参考：

### 方案 A：最稳妥
- 全局挂载：`web/src/pages/pollo.ai/_layout/layouts/RootLayout/index.tsx`
- 开始入口：`web/src/utils/generation-task-link.ts`
- 轮询结构：`web/src/pages/pollo.ai/home/canvas/_hooks/useCanvasPolling.ts`
- 状态接口：`server/router/routes/generation.ts -> queryRecordDetail`

适合：
- 先做一版稳定功能
- 需要同时支持 image/video
- 完成后弹提醒

### 方案 B：批量优化
- 全局挂载：`RootLayout` 或 `_app.page.tsx`
- 开始入口：`generationTaskLink`
- 队列持久化：`Header/History`
- 轮询接口：`video.getVideoStatusByRecordIds`

适合：
- 需要同时轮询多个任务
- 更关注性能和批量状态刷新

### 方案 C：调度器化
- 全局挂载：`RootLayout`
- 容器结构：仿 `AgentScheduler`
- 开始入口：`generationTaskLink`
- 轮询逻辑：仿 `useCanvasPolling`
- 查询接口：先 `getVideoStatusByRecordIds`，必要时补 `queryRecordDetail`

适合：
- 未来会持续扩展
- 想把“注册任务 / 取消任务 / 完成回调 / 去重 / 间隔控制”都收敛成基础设施

---

## 结论

现有项目里已经具备完整拼图，但分散在不同模块：

- 全局入口：`generationTaskLink`
- 全局挂载模式：`_app.page.tsx` / `RootLayout`
- 页面级轮询：`useRecordsStatusMonitor`
- 全局单点轮询参考：`useCanvasPolling`
- 队列持久化参考：`Header/History`
- 调度器抽象参考：`AgentScheduler`
- 通用状态查询：`generation.queryRecordDetail`
- 批量状态查询：`video.getVideoStatusByRecordIds`

如果后续正式落地这个功能，建议优先复用：
- `generationTaskLink`
- `RootLayout`
- `useCanvasPolling` 的结构
- `queryRecordDetail / getVideoStatusByRecordIds`
